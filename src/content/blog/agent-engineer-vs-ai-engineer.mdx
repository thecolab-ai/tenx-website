An agent engineer is an AI engineer who specialises in autonomous multi-step systems — the category of AI builds where a language model selects tools, interprets their results, and takes further actions with limited human intervention per cycle. The specialisation is not a different discipline from AI engineering; it is a depth investment in the part of the AI stack that is hardest to evaluate, hardest to make reliable, and fastest-growing in 2026. Whether you need an agent specialist or a broader AI engineer depends on the scope and reliability requirements of your autonomous system.

## What agent engineers actually build

The output of an agent engineer is not a model and not a single prompt — it is a system that can act. The agent itself may be simple or complex, but the engineering work is always the same category: designing something that takes actions you cannot fully enumerate in advance.

The work covers:

- **Tool design**: the interface between the model and the external world. Every tool an agent can call has a schema — a description, an argument definition, an expected return format. The schema is the engineering artifact that most shapes agent reliability. A schema that is too permissive lets the model hallucinate arguments. A schema that is too strict breaks on edge cases. Getting tool schemas right requires understanding both the model's interpretation behaviour and the downstream system's constraints.
- **Loop control**: the logic that decides when an agent continues, when it stops, and when it escalates. A loop without a budget or a circuit-breaker runs until it fails expensively. An agent engineer designs the budget (step count, token count, wall-clock time, cost), the termination conditions, and the escalation paths — what happens when the agent is stuck, when it produces a low-confidence result, or when an action is irreversible.
- **Failure recovery**: the handling of tool errors, unexpected outputs, and model refusals inside a running loop. A production agent fails regularly — API timeouts, malformed returns, edge inputs — and must handle them gracefully. This means retry logic, fallback strategies, partial-result handling, and state management across a loop that can span many steps.
- **Human override paths**: the mechanism for a human to review, correct, or halt an agent action before it executes. The engineering question is where the override sits, how it is triggered, and how it is surfaced to the right person at the right time. An agent that cannot be interrupted is a production liability.
- **Observability**: the logging and tracing infrastructure that makes an agent's reasoning visible. Unlike a request-response API where debugging starts with the input and output, a multi-step agent requires step-by-step traces — what the model decided at each step, what tool it called, what the tool returned, how it interpreted the result. Without this, production failures are invisible.
- **Evaluation**: measuring whether the agent is working. Agent evals are qualitatively different from model or prompt evals. The output of a single agent run is a sequence of actions, not a single answer, and the same correct outcome can be reached via different paths. A good agent eval must assess intermediate steps, not just final results, and must cover the failure modes that matter — loop failures, tool misuse, escalation timing — not just success cases.

An agent engineer's output is a system that acts reliably on the kinds of inputs it will encounter in production, fails gracefully on the ones it cannot handle, and can be observed and corrected when its behaviour drifts.

## How agent engineering fits inside AI engineering

AI engineering covers model fluency, prompt architecture, retrieval systems, evals infrastructure, and operational monitoring. Agent design is one layer of that surface — the layer concerned with autonomous multi-step behaviour. The rest of the surface is still required. An agent engineer needs Claude, OpenAI, and Google fluency (different models behave differently in agentic loops), prompt engineering depth (tool descriptions are a form of prompt), and evals expertise (agent evals are harder to design than single-output evals, not easier).

What separates an agent specialist from a generalist AI engineer is the depth of investment in the specific failure modes of autonomous systems. Every AI engineer can build a basic agentic workflow — call a tool, interpret the result, call the next tool. What the generalist does not typically have is the mental model of what breaks when the loop runs for thirty steps on an input that is slightly outside the training distribution, or the tooling to observe and diagnose that failure, or the eval structure that would have caught it before it reached production.

## When to hire an agent specialist vs a broader AI engineer

Most companies in 2026 building their first agentic feature do not need an agent specialist. They need a capable AI engineer who has built agent systems as part of a broader portfolio. The specialist is the right hire when the agent itself is the product problem — when reliability, traceability, and oversight of an autonomous system are the core engineering challenge, not one component among many.

Hire an agent specialist when:

- **The agent is customer-facing and must operate reliably at scale.** Customer support agents, research synthesis tools, code review systems — any agent where a failure is visible to a user and a runaway loop is costly in dollars or reputation. At that level, the agent's reliability is the product, and a specialist moves faster.
- **The system involves irreversible actions.** Agents that send emails, execute code, make API calls with side effects, or modify state need careful override paths and audit logs. An engineer with production experience in irreversible agent actions has built these before.
- **You already have an agent in production that is underperforming.** Runaway loops, hallucinated tool calls, poor escalation timing, and invisible failures are common in first-generation production agents. A specialist can diagnose these faster than a generalist because they have seen the failure modes before.
- **The loop is long and the state space is complex.** A two-step agent is a prompt. A twenty-step agent with branching paths, external tool calls, and partial-result handling is a systems engineering problem. Complexity scales faster than intuition suggests.

A broader AI engineer with an agent design badge is the right hire when:

- You are building multiple AI features, of which agents are one. RAG, evals, model routing, and product integration are all in scope alongside agent workflows.
- The agent is bounded — a specific, time-limited capability within a larger product, not the product itself.
- You need someone to own the full AI layer, not just the autonomous system component.

In practice, most companies hiring in Auckland, Wellington, Sydney, and Melbourne in 2026 are at the stage where the broader AI engineer is the right hire. The agent specialist becomes valuable when a team has already built and shipped agents and is hitting reliability and observability limits that generalist experience cannot solve quickly.

## What separates good agent engineering from bad

The gap is almost always observability and the loop contract.

A weak agent implementation produces a system that works in demos. Tool calls succeed, the loop terminates, the output looks plausible. What it lacks is the infrastructure to know when it stops working — and when an agent system fails silently in production, it fails in ways that are harder to trace than a straightforward API call.

Strong agent engineering starts with the loop contract: a precise definition of what the agent is allowed to do, how long it may run, what constitutes a successful termination, and when it must escalate. This contract is written before the implementation, not discovered by running the agent and observing what it does. Writing the contract forces the engineer to answer the questions that production will ask: what happens when a tool returns a malformed result? What happens when the model calls a tool with an argument that is syntactically valid but semantically wrong? What happens when the task is ambiguous?

After the loop contract comes the observability infrastructure. Every step in the loop is logged — the model's reasoning, the tool it called, the arguments it passed, the result it received, the next decision it made. This trace is the primary debugging artifact for a production agent. Without it, a failure twenty steps deep in a long-running agent is nearly impossible to diagnose.

After observability comes evaluation. A strong agent eval covers:

- **Step-level correctness**: at each step in the loop, did the agent do the right thing? This requires a test set with annotated correct paths — not just correct final outputs.
- **Failure mode coverage**: does the eval include cases where the agent should escalate, where it should terminate early, where a tool call should fail? A test set that only covers success paths cannot catch the failure modes that matter most in production.
- **Regression detection**: when the underlying model is updated or the prompt changes, does agent behaviour change in ways that matter? This requires a baseline trace for a representative set of inputs and a mechanism to compare against it.

The specific failure cases that appear in production are the most instructive. An agent specialist hired to stabilise an underperforming system typically finds: the tool schemas were written to be permissive to avoid build friction and now produce a high rate of hallucinated arguments; the loop has no budget and occasionally runs for hundreds of steps before hitting a platform limit; and there is no step-level logging, so failures are only visible in the final output, if at all. These are standard first-generation production agent problems. They are solved by the loop contract, the observability layer, and the eval harness — in that order.

- **Tool schema design**: candidates are given a task specification and asked to design the tool schemas an agent would use, including argument definitions, type constraints, and descriptions. ## How to hire the right person

Start by deciding whether you need an agent specialist or a broader AI engineer with agent depth. Most hiring teams in 2026 need the latter, but being specific about this before sourcing saves significant time.

If you need agent depth, structure your assessment around production questions:

- Can the candidate describe an agent failure mode they diagnosed in a live system? Not a theoretical failure — a real one, with a specific tool schema, a specific symptom, and a specific fix.
- Can they describe the loop contract for a system they built? Specifically: what was the budget, what were the termination conditions, what constituted an escalation trigger, and how was that contract enforced?
- Can they walk through the eval structure for a production agent? What did they test at step level, what failure modes did they include in the test set, and how did they detect regressions?

A candidate who answers these with specifics — concrete tool schemas, real failure modes, actual eval structures — has production agent depth. Generic answers about agentic patterns indicate familiarity with the concept, not engineering experience with the problem.
